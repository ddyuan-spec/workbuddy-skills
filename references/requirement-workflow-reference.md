# 泰小虎需求输出 workflow 参考手册

## 一、项目信息必问清单（Step 1）

### 1.1 端口与系统
- 是否有管理后台？有几个？分别怎么命名？
- 是否有客户端？有几个？分别是什么？怎么命名？
- 是否有 H5 / 小程序 / App（Android/iOS）？
- 本期需求是否涉及后端服务？

### 1.2 设计参考
- 每个端口是否有现有设计规范或组件库？
- 是否有参考截图 / HTML / 墨刀 / Figma 文件？
- 若找不到合适参考，必须追问用户补充，禁止凭空设计。

### 1.3 历史信息与数据
- 是否有已上线功能需要兼容？
- 是否有历史数据需要迁移？
- 是否关联其他已做需求？关联方式是什么？
- 是否有通用规则（如埋点规范、用户等级规则）需要沿用？

### 1.4 人员与排期
- 产品负责人（PM）
- 研发负责人（前端 / 后端 / H5）
- 测试负责人
- UED / 交互
- 数据 / 埋点对接人
- 预计里程碑

---

## 二、需求梳理阶段交付物（Step 2）

### 2.1 背景与目标
- 业务背景：为什么做？
- 业务目标：提升什么指标？
- 用户目标：用户获得什么价值？
- KPI / 成功指标：可量化、可追踪。

### 2.2 已确认关键决策
- 用表格列出所有关键决策点与结论。
- 决策点示例：定位、券种、叠加规则、底价规则、适用范围、发放方式、退券规则、到期提醒等。

### 2.3 影响范围
- 对老版本 / 已上线功能的影响。
- 对关联需求的影响。
- 对数据 / ERP / 消息中心 / 埋点等中台能力的影响。

### 2.4 业务流程图
- 必须包含主流程 + 异常/分支流程。
- 必须为常用流程图样式（SVG / 流程图图片），不能是 Markdown 格式。

### 2.5 信息架构
- 清晰界定前端展示层 / 后端配置层 / 数据层的映射关系。
- 必须为图片 / SVG，不能是 Markdown 格式。

### 2.6 状态机与状态流转规则
- 凡涉及状态变化的实体，必须提供状态流转表。
- 格式：起始状态 + 触发动作(Action) + 前置/后置条件 = 目标状态。

### 2.7 用户旅程 / 脑图
- 根据业务流程和用户生命周期，关联板块确认功能范围。
- 脑图优先输出 Markdown 大纲（可导入钉钉文档切换为思维导图）。

### 2.8 统一查看文件
- 以上所有梳理内容生成一个 HTML 文件，统一给用户确认。
- 文件命名：`{需求名}-需求梳理图集.html`
- 该文件保留到需求文档中，图将用于后续 PRD 书写。

---

## 三、原型输出要求（Step 3）

### 3.1 每个端口独立原型
- 根据 Step 1 了解的端口，每个端口输出可交互 HTML 原型。
- 管理端原型：列表页 / 表单页 / 弹窗 / 状态切换。
- 客户端原型：页面流程 / 空态 / 异常态 / 弹窗。

### 3.2 原型确认清单
每次交付原型时必须确认：
- 页面是否完整？
- 弹窗是否完整？
- 交互逻辑是否完整？
- 异常分支是否覆盖？
- 字段/按钮/权限是否完整？

---

## 四、PRD 输出要求（Step 4-6）

### 4.1 PRD 结构检查（强制）
- 在输出 PRD 前，必须先调用 `prd-structure-checker` skill 做结构排查。
- PRD 必含板块只能多不能少。
- §D 功能需求详情必须按《物流费用账单 V2.10.3》模板逐功能点细写。

### 4.2 PRD 输出格式
- 固定输出为本地 Word 文档（.docx）。
- 图片必须正确渲染，且为之前生成的图片格式（SVG / PNG），不能是 Markdown 格式文件。
- PRD 中不得包含待确认标识和冗余信息。
- 所有待确认信息另外输出一个本地文档：`{需求名}-待确认项.docx`。

### 4.3 PRD 内容模块
必须包含（但不限于）：
1. 文档基本信息与修订记录
2. 背景、目标与 KPI
3. 已确认关键决策
4. 功能需求清单（FR）
5. 核心业务规则与限制
6. 业务流程与状态机
7. 需求影响范围
8. 涉及终端明细
9. 平台管理端功能需求详清
10. 客户端功能需求详清
11. 验收标准（AC）
12. 埋点需求
13. 非功能性需求
14. 边界设计检查清单
15. 关联需求
16. 待确认 / 开放问题

### 4.4 反问 Loop
- PRD 输出后进入循环确认。
- 颗粒度由粗到细：先反问流程维度，再反问细节如字段处理。
- 每个循环有问题都要积极反问，细化到字段级别。

---

## 五、GitHub 存档与评审后修改（Step 7-8）

### 5.1 首次存档
- 终版原型和终版 PRD 推送到 GitHub 等待评审。
- 推送前确保：
  - 仅包含结果文件
  - 过程文件已删除
  - 生成器脚本不上 GitHub

### 5.2 评审后修改
- 线下评审后，根据反馈下载 GitHub 文件到本地修改。
- 用户确认后再重新推送 GitHub 更新。
- 每次更新需记录修订记录。

---

## 六、收尾（Step 9）

### 6.1 文件整理
- 终版文件保存。
- 其他过程文件删除。
- 本地文件归入对应需求文件夹。
- 生成器脚本随源文件同目录保存，但不上 GitHub。

### 6.2 文档索引更新
- 更新 `00-文档索引.html`，加入最新产出链接。

### 6.3 记忆更新
- 更新项目 MEMORY.md 或每日日志。

---

## 七、文件命名规范

### 7.1 需求框架
`{需求名}-需求框架.html`

### 7.2 需求梳理图集
`{需求名}-需求梳理图集.html`

### 7.3 原型
- 单页原型：`{需求名}-{模块}-{页面}.html`
- 合并交互原型：`{需求名}-{模块}-合并交互原型.html`
- 平铺稿：`{需求名}-{页面}-flat.html`

### 7.4 PRD
`{需求名}-PRD.docx`

### 7.5 待确认项
`{需求名}-待确认项.docx`

---

## 八、关键约束

1. **禁止臆测**：不确定数据必须 @干系人 占位或用 AskUserQuestion 反问。
2. **按需求分块**：每个需求独立文件夹，禁止跨需求文件夹引用/依赖。
3. **每次只展示最新产出**：present_files 只展示最新定版文件，不堆历史版本。
4. **PRD 先过结构检查**：所有 PRD/需求文档必须先调用 prd-structure-checker。
5. **图片格式正确**：PRD 中的图必须是图片格式，不能是 Markdown 流程图。
