# PRD 终稿结构规范（对齐《产品需求文档（PRD）输出规范与标准示例（修订于260715）》）

> **本文件就是桌面那份《产品需求文档（PRD）输出规范与标准示例（修订于260715）》的结构权威。**
> 权威源文件：`C:/Users/13364/Desktop/产品需求文档（PRD）输出规范与标准示例 （修订于260715）.docx`
> 所有新功能需求文档 / PRD 终稿**必须严格照此「一~九」骨架 + 页面式 §四 写法**产出，
> `scripts/check_prd.py` 也按此骨架做存在性 + 深度扫描。
> 原则：**只能多不能少**——终稿必须覆盖下列全部板块，缺失任一板块即不通过。

---

## 规范宗旨（三大原则）

1. **宁多勿漏**：宁可在设计阶段写得详细，也不在开发阶段补丁重重。
2. **多端联动**：凡涉及用户、运营、商家的功能，必须全链路闭环，严禁"断头需求"。
3. **凡变必录**：评审后的任何需求变更，必须有版本记录并高亮通知干系人。

## 终稿必含板块清单（一 ~ 九，缺一不可）

### 一、文档基本信息（元信息）
- 项目名称
- 创建时间
- 负责人（PM / 技术 Owner / 测试负责人）
- 文档状态（草稿 / 评审中 / 已定稿 / 变更中）
- **历史修订记录（表格）**：版本号 / 修订日期 / 修订人 / 修订内容简述 / 状态

### 二、项目背景与目标
- **业务背景（痛点分析）**：明确指出真实痛点，必须用数据或具体场景说话，拒绝空洞词汇。
- **项目目标**：包含业务目标与用户目标。
- **衡量指标（KPI）/ 埋点数据**：明确上线后看哪些指标。

### 三、业务流程与架构
- **1. 业务流程图**：必须包含主流程与异常/分支流程（建议用图，标注平台端 / 用户端 / 异常 / 判断节点）。
- **2. 信息架构**：清晰界定前端展示层、后端配置层、数据层的映射关系。
- **3. 状态机与状态流转规则**：凡涉及状态变化的实体（订单、券、推送等），必须提供状态流转表，定义 `起始状态 + 触发动作(Action) + 前置/后置条件 = 目标状态`。

### 四、功能需求详情（**按页面拆解**，这是本模板与他模板最核心差异）
> **每个页面 = `X. 页面：<页面名>`**，下含两个固定子节：
> - **X.1 页面原型与交互示意**：该页面一张原型截图（图文对得上，禁止把流程 SVG 当界面截图）+ 交互说明（点击/滑动/弹窗/遮罩等手势与动态反馈）。
> - **X.2 详细逻辑规则**，下含两个固定子节：
>   - **界面与展示规则**：列表字段表（字段名称+需求说明+备注，单列排序/分页/合计/数据唯一判断）、功能按钮权限表（功能说明+显示位置+权限控制）、展示样式与空/异常态。
>   - **业务逻辑与边界条件**：单个按钮四要素 + 逐步操作逻辑（功能说明/显示位置/权限/前置条件/功能实现含异常分支+校验提示文案）、编辑/新增表单规则（输入限制 必填/类型/长度/小数位 + 计算公式）、查询条件规格表、数据来源字段级标注（前端本地/后端数据库/第三方接口，或「取《XX表》」「调XX接口」）、计算公式+数据演练、边界条件（断网超时/登录拦截/未授权/防抖防重/并发防刷）。

### 五、非功能性需求
- 性能、兼容性、安全、埋点要求等。
- **埋点需求（链接引用，禁止内嵌整表）**：PRD §五 埋点需求**不得把《泰小虎埋点表v2.3》整表复制进文档**（30 事件 × 属性会极难看且臃肿）。标准写法三件套：
  1. 一句说明：「本需求埋点事件与属性严格遵循《泰小虎埋点规范 v2.3》」；
  2. 插入链接：`[泰小虎埋点规范 v2.3（在线查阅）](https://ddyuan-spec.github.io/taixiaohu/taixiaohu-tracking-spec.html)`（完整 30 事件 / 属性 / 页面枚举 / 友盟约束在线全文，读者点开查阅）；
  3. **本期涉及事件清单**（精简表，仅列本期用到）：`事件英文名 | 事件显示名 | 所属层 | 平台 | 触发时机 | 必含上报参数(友盟中文名) | 备注`。不展开全部属性列、不列未涉及事件。
- **埋点规范强制引用**：埋点事件与属性**必须从《泰小虎埋点表v2.3》既有目录选取**（AI 写作用本 skill `references/tracking-spec.md` 结构化参考；权威源文件 `D:/梯子/泰小虎埋点表v2.3 (5).xlsx`；在线方案 https://ddyuan-spec.github.io/taixiaohu/taixiaohu-tracking-plan.html ）。**禁止凭空编造事件英文名 / 属性变量名**；新事件须走埋点评审，不在 PRD 内新增定义。引用埋点参数一律用友盟后台**中文名**（如「步骤序号」而非 step_index）。

### 六、验收标准
- 站在测试（QA）角度写出关键场景验收标准（AC）。
- 推荐采用 `Given（前提） -> When（操作） -> Then（结果）` 描述。

### 七、关键边界设计检查清单（PM 撰写前自检表）
逐项自检并保留勾选结果：
- 网络与异常状态（弱网 / 断网 / 服务器 500 时页面与按钮状态）
- 登录与游客态（未登录点击核心按钮的处理；登录成功后是否自动继续刚才的操作）
- 极限值与空状态（空列表/缺省图与文案；输入字符上限与文本超长展示；价格/金额展示极限）
- 版本兼容性（老版本看到新功能的表现；新旧数据字段兼容）
- 逆向流 / 退款（退款、退货、取消时系统处理）
- 高并发与防重（提交/支付/领券 是否做前端防重抖）

### 八、文档维护与变更规范
- 版本号规范（大迭代 V1.0/V2.0；小修改 V1.1/V1.2，历史记录写明修改人/时间/原因）
- 变更周知（定稿后变更须在文档中高亮标出，并主动 @ 对应开发和测试）

### 九、待确认项汇总
- 全文「⚠️ 待确认 @干系人」占位项集中罗列，便于评审时逐条澄清。

## 不确定性处理（红线，写 §四 前必做）
写 §四 时，**先逐页面/逐功能点梳理「已知 vs 未知」**。凡属以下任一情况，即视为「不确定数据」，**禁止由模型臆测填充**：
- 字段长度 / 输入限制 / 格式未指定；
- 按钮的权限角色 / 数据范围未定义；
- 计算公式 / 状态流转未明确；
- 字段来源（前端 / 后端 / 第三方）不清楚；
- 异常提示文案、边界值未给；
- 任何「我觉得应该是…」「先这样占位」的假设。

处理方式（二选一或并用）：
1. **文档内占位**：以「⚠️ 待确认 @干系人」明确标注，绝不编造具体内容；
2. **主动反问**：用 `AskUserQuestion` 向用户反问**具体**问题（如「XX 字段的输入长度限制是多少？」），拿到确切答案后再写。

写完后必须向用户汇总「本次有 N 处待确认项，已标注 / 已提问」，不可静默带过。若用户坚持先占位：只允许「⚠️ 待确认 @干系人」占位，**禁止编数值**。

> **排查联动**：`scripts/check_prd.py` 在「一~九」存在性扫描之后，会对 §四 做页面式深度子项扫描（界面与展示规则 / 业务逻辑与边界条件 / 原型截图等）；并在阶段3 扫描全文「待确认 / 需确认 / @X确认 / TBD / 待补充」等标记，列出所有待确认项。

---

## 排查判定口径
- **完整 ✅**：板块存在且包含上述关键子项，内容非占位/非空。
- **单薄 ⚠️**：板块存在但缺失关键子项，或内容明显为空泛占位（如"待补充""TBD"）。
- **缺失 ❌**：终稿中完全找不到该板块或其等价表述。

任一 **缺失** 或 **单薄** 项，都必须向用户反问补充，不得由模型臆造内容。
