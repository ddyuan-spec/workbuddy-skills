# PRD 结构排查器（prd-structure-checker）

## 用途

对照团队《产品需求文档（PRD）输出规范与标准示例（修订于260715）》的骨架，
对一份 PRD / 需求文档终稿做**结构完整性排查**。核心规则只有一条：
**必含板块只能多不能少**——终稿必须覆盖规范中的全部板块（一~九），缺失任一即不通过；
额外板块允许存在。凡是缺失、单薄或未知的内容，**必须反问用户补充，不得由模型凭空生成**。

本 skill 是**全局强制规则**：任何任务只要产出/审阅/修订“需求文档 / PRD”，
都必须先调用本 skill 完成结构排查，通过后才能定稿或进入评审。

## 何时使用

- 用户要求写、改、审一份 PRD / 需求文档 / 产品需求说明书。
- 用户说“帮我出 PRD”“润色需求文档”“检查这份 PRD 缺不缺东西”“终稿排查”。
- 任何以“需求文档”“功能需求”“产品方案”为交付物的任务收尾阶段。
- 即在对话中未明说，只要产物是 PRD 类文档，也强制触发本排查。

## 排查流程

1. **加载规范依据**：读取 `references/prd-structure.md`，掌握必含板块 一~九 及其关键子项。该文件是判定唯一依据。
2. **获取终稿**：接收 PRD 文件路径或全文内容。若用户在对话中直接给出文本，先落盘为 `.md` 再排查。
3. **跑结构扫描（两阶段）**：用 `scripts/check_prd.py` 对文档做确定性扫描。
   ```bash
   python scripts/check_prd.py 路径/到/PRD文档
   ```
   - **阶段1（结构存在性）**：一~九 必含板块是否出现，缺失即不通过。
   - **阶段2（§四 内容深度）**：对功能需求详情做"模板式详清"扫描，逐项判定
     `需求影响范围 / 涉及终端明细 / 界面设计·原型截图 / 交互说明 / 字段明细表 /
     功能按钮权限表 / 逐步操作逻辑 / 查询条件规格 / 业务逻辑·计算规则 / 边界条件 / 数据来源标注`
     是否齐备；缺失超过阈值（默认 3 项）判为 **⚠️ 结构通过但深度单薄**，并列出缺失子项。
     这一步能拦住"扁平一行摘要表"蒙混过关（详见 `references/prd-structure.md` 的 D 板块详清规范）。
   将脚本输出作为初判，再结合人工（模型）判断修正“单薄 ⚠️”项。
4. **逐板块定级**：对照 `references/prd-structure.md` 把每个板块定为三类之一：
   - ✅ 完整：板块存在且含关键子项，内容非占位。
   - ⚠️ 单薄：板块存在但缺关键子项，或内容为空泛占位（如“待补充”“TBD”“详见xxx”无实质）。
   - ❌ 缺失：终稿完全找不到该板块或其等价表述。
5. **强制反问补充（核心，不可跳过）**：对每一个 ❌ 缺失 或 ⚠️ 单薄 的板块：
   - **禁止臆造**：不得替用户编内容填空。
   - **直接反问**：用 `AskUserQuestion` 或清晰的追问，列出该板块“具体缺什么”，请用户补充。
   - 一次性可合并多个缺失项提问，但每项都要指明缺的内容（如“缺‘状态机与状态流转规则’：请补充涉及状态变化的实体及其流转表”）。
   - 用户补充或**明确声明该项不适用/可省略**后，才将该项标记为已处理；用户未确认前不得定稿。
6. **输出排查报告**：按以下格式汇总（可直接呈现给用户）：

   ```
   # PRD 结构排查报告
   文档：（文档名）
   结论：通过 / 不通过（缺失 N 项）/ ⚠️ 结构通过但 §四 深度单薄

   ## 阶段1：一~九 必含板块存在性
   | 必含板块 | 状态 | 命中关键词 |
   | 一、文档基本信息 | ✅ | 含修订记录表 |
   | 四、功能需求详情 | ✅ | 功能需求详情 |
   | ... | ... | ... |

   ## 阶段2：§四 功能需求详情 — 内容深度检查
   | 深度子项 | 状态 | 命中关键词 |
   | 页面原型与交互示意 | ✅ | 原型、figure |
   | 界面与展示规则(字段表/按钮权限) | ✅ | 字段名称、权限 |
   | 业务逻辑与边界条件(逐步操作逻辑) | ❌ 缺失 | — |
   | ... | ... | ... |

   待补充（已向你反问）：
   - 三、业务流程与架构·状态机：请补充订单/券的状态流转（起始状态+动作+条件=目标状态）
   - §四 逐步操作逻辑：请补充各功能的"前置条件→步骤→异常分支"
   - 七、关键边界设计检查清单：未找到逆向流/退款自检，请补充
   ```

7. **定稿门槛**：仅当所有必含板块均为 ✅（或用户明确豁免）时，方可判定“通过”并允许定稿/进入评审。

## 输出风格：简洁优先

本 skill 要求结构完整、字段细化、不确定性明确标注，但**不要求冗长**。写 PRD 时应遵循：

1. **能删则删**：删除与当前功能无关的背景铺垫、重复解释、过度修饰的过渡句。背景控制在 1-2 段，目标/指标用表格逐行呈现。
2. **用什么 + 条件/规则即可**：每个功能小节以「做什么」和「有什么条件/规则」为核心，避免大段叙事。逐步逻辑用编号步骤，异常分支用 bullet。
3. **表格承载细节**：字段映射、验收标准、边界条件、权限、埋点等尽量用表格，减少散文式描述。
4. **不确定项集中标注**：用「⚠️ 待确认 @干系人」一句话占位，不要展开解释为什么不确定。
5. **禁止为凑篇幅而扩写**：结构板块只能多不能少，但每个板块内部内容应精简到可开发/可评审所需的最小信息量。
6. **纯后端/接口契约 PRD 的例外**：若 PRD 明确不涉及前端界面（如仅定义后端接口、数据交互规则、Webhook 回调），可在 §四 中用一个简短「范围声明」列出不适用的前端深度子项（如界面设计/原型截图、交互说明、前端字段明细表、功能按钮权限表），并注明由哪份既有需求覆盖。结构排查时这些关键词会被命中，视为已声明。

## 关键边界（来自规范宗旨）

- 宁多勿漏：宁可多写，不可漏板块。
- 多端联动：涉及用户/运营/商家须全链路闭环，警惕“断头需求”。
- 凡变必录：变更须有版本记录并高亮 @ 干系人。

## §四 细节书写标准（强制套用《物流费用账单 V2.10.3》模板）

写 PRD 的 **§四 功能需求详情** 时，不允许写一段扁平摘要，必须按 `references/prd-detail-template.md` 的范本**逐功能点**拆解。要点：

1. **逐功能点拆分**：以业务对象为顶层，向下拆成「状态流转 → 权限 → 列表 → 功能按钮总表 → 每个按钮独立小节 → 编辑/新增表单 → 查询条件 → 菜单权限」。**一个按钮 = 一个独立小节**。
2. **列表字段表**：字段名称 / 需求说明 / 备注，并单列排序、分页、合计行、数据唯一判断。
3. **按钮四要素 + 逐步逻辑**：每个按钮写「功能说明 / 显示位置 / 权限 / 前置条件 / 功能实现」，功能实现必须含逐步逻辑 + 异常分支 + **校验提示文案**（如未勾选→“请选择数据”、勾选已确认→“仅未确认支持确认”）。
4. **编辑/新增表单**：按区域分组（区域/字段/功能逻辑说明/备注），每个字段写明输入限制（必填/类型/长度/小数位），并列出**计算公式**。
5. **查询条件**：查询条件 / 查询说明（控件类型 / 选项来源 / 查询行为）。
6. **字段级数据来源**：列表字段须标注「取《XX表》字段」「调 XX 接口」，区分前端本地 / 后端 / 第三方。
7. **计算公式 + 数据演练**：复杂计算须附具体数字演练并保证合计守恒。
8. **逐页截图**：每个页面一张图（列表/编辑/弹窗），图文对得上；禁止用流程 SVG 充当界面截图。

> 阶段2 深度扫描（见 `scripts/check_prd.py` 的 `DEPTH_ITEMS`）即按上述 8 项判单薄；
> 缺失超 3 项 → 结论判为「⚠️ §四 深度单薄」并列出缺失项，触发向用户反问补充。

## 不确定性处理与反问机制（核心红线，不可跳过）

写 §四（以及任何板块）遇到不确定数据时，**第一反应不是编，而是问或标**。

**写之前**：先逐页面/逐功能点梳理「已知 vs 未知」，列出所有不确定点。凡属以下任一，即视为「不确定数据」，禁止臆测：
- 字段长度 / 输入限制 / 格式未指定；
- 按钮权限角色 / 数据范围未定义；
- 计算公式 / 分摊 / 状态流转未明确；
- 字段来源（前端 / 后端 / 第三方）不清楚；
- 异常提示文案、边界值未给；
- 任何「我觉得应该是…」「先这样占位」的假设。

**处理方式（二选一或并用）**：
1. **文档内占位**：以「⚠️ 待确认 @干系人」明确标注（模仿模板「具体逻辑需 @Lven 确认」），绝不编造内容填进去；
2. **主动反问**：用 `AskUserQuestion` 反问**具体、可落地**的问题（例：「XX 字段的输入长度限制是多少？」「未确认状态点击删除的提示文案是什么？」），拿到确切答案后再写。

**写之后**：必须向用户汇总「本次有 N 处待确认项，已标注/已提问」，不可静默带过。
若用户坚持先占位：只允许「⚠️ 待确认 @干系人」占位，**禁止编数值**。
阶段3 扫描（见脚本）会检测全文「待确认 / 需确认 / @X确认 / TBD / 待补充」等标记并列出，提醒澄清或保留占位。

## 文件说明

- `references/prd-structure.md`：必含板块 一~九 清单与判定口径（唯一依据），含 D 板块深度门槛与不确定性红线。
- `references/prd-detail-template.md`：§四 细节书写范本（逐功能点拆解、字段表、按钮分步逻辑、计算公式、字段级数据来源、数据演练）。
- `scripts/check_prd.py`：结构扫描脚本，支持 .md/.txt/.docx/.html，三阶段输出（A~I 存在性 / §四 深度 / 待确认项）。
