# 需求输出 Workflow 参考手册（通用）

> 适用于任意项目。本手册为 `requirement-workflow` skill 的详细参考，与其 10 步流程一一对应。

## 一、项目信息必问清单（Step 1）

> ⚠️ **引导式收集（对应总体原则第 10 条）**：以下清单**不要一次性抛给用户**。AI 按「引导顺序」逐轮只问一个维度（用单问题 AskUserQuestion 或文字），用户答完再问下一个。**端口最高优先、必须最先单独对齐**，其余依次推进。

**引导顺序**：① 设计参考样式 → ② 历史信息/数据 → ③ 关联需求 → ④ 人员（逐人）→ ⑤ 里程碑。

### 1.1 端口与系统
- 是否有管理后台？有几个？分别怎么命名？
- 是否有客户端？有几个？分别是什么？怎么命名？
- 是否有 H5 / 小程序 / App（Android/iOS）？
- 本期需求是否涉及后端服务？

### 1.2 设计参考
- 每个端口是否有现有设计规范或组件库？
- 是否有参考截图 / HTML / 墨刀 / Figma 文件？
- 若找不到合适参考，必须追问用户补充，禁止凭空设计。

### 1.3 历史信息与数据
- 是否有已上线功能需要兼容？
- 是否有历史数据需要迁移？
- 是否关联其他已做需求？关联方式是什么？
- 是否有通用规则（如埋点规范、用户等级规则、价格规则）需要沿用？

### 1.4 人员与排期（逐人引导，里程碑单独一轮）
- 产品负责人（PM）—— 先问这一个角色
- 研发负责人（前端 / 后端 / H5 / 客户端）—— 用户给完 PM 再问
- 测试负责人
- UI / 交互
- 数据 / 埋点对接人
- 预计里程碑 —— 最后单独一轮问，不与其他人员混问

---

## 二、需求梳理阶段交付物（Step 2）

### 2.1 背景与目标
- 业务背景：为什么做？
- 业务目标：提升什么指标？
- 用户目标：用户获得什么价值？
- KPI / 成功指标：可量化、可追踪。

### 2.2 已确认关键决策
- 用表格列出所有关键决策点与结论。
- 决策点示例：定位、核心对象、叠加/互斥规则、价格/底价规则、适用范围、发放/触发方式、退出/退款规则、到期提醒等。

### 2.3 影响范围
- 对老版本 / 已上线功能的影响。
- 对关联需求的影响。
- 对数据 / 中台能力 / 消息中心 / 埋点等的影响。

### 2.4 业务流程图
- 必须包含主流程 + 异常/分支流程。
- 必须为常用流程图样式（SVG / 流程图图片），不能是 Markdown 格式。

### 2.5 信息架构
- 清晰界定前端展示层 / 后端配置层 / 数据层的映射关系。
- 必须为图片 / SVG，不能是 Markdown 格式。

### 2.6 状态机与状态流转规则
- 凡涉及状态变化的实体，必须提供状态流转表。
- 格式：起始状态 + 触发动作(Action) + 前置/后置条件 = 目标状态。

### 2.7 用户旅程 / 脑图
- 根据业务流程和用户生命周期，关联板块确认功能范围。
- 脑图优先输出 Markdown 大纲（可导入钉钉文档切换为思维导图）。

### 2.8 统一查看文件
- 以上所有梳理内容生成一个 HTML 文件，统一给用户确认。
- 文件命名：`{需求名}-需求梳理图集.html`
- 该文件保留到需求文档中，图将用于后续 PRD 书写。

## 三、原型输出要求（Step 3）

### 3.1 每个端口独立原型
- 根据 Step 1 了解的端口，每个端口输出可交互 HTML 原型。
- 管理端原型：列表页 / 表单页 / 弹窗 / 状态切换。
- 客户端原型：页面流程 / 空态 / 异常态 / 弹窗。

### 3.2 原型确认清单
每次交付原型时必须确认：
- 页面是否完整？
- 弹窗是否完整？
- 交互逻辑是否完整？
- 异常分支是否覆盖？
- 字段/按钮/权限是否完整？

### 3.3 客户端设计原型（平铺稿，墨刀用）【必做】
可交互原型确认后，**仅针对客户端**（App / H5 / 小程序，不含管理后台）额外输出一份平铺设计原型：
- 所有客户端页面按顺序平铺在一个 HTML 里，**无交互、无跳转、无弹窗逻辑**，纯静态一屏总览。
- 页面提供「一键复制源码」按钮，点击把完整 HTML 源码复制到剪贴板，便于粘贴到墨刀。
- 每页配简短标题标注，不做任何交互控件。
- 命名：`{需求名}-客户端设计原型-flat.html`。

### 3.4 原型图片校验
原型/平铺稿引用图片时，生成后按「图片校验 Loop」确认本地渲染正常，推送后确认远端渲染正常。

---

## 四、PRD 输出要求（Step 4-6）

### 4.1 PRD 结构检查（强制）
- 在输出 PRD 前，必须先调用 `prd-structure-checker` skill 做结构排查。
- PRD 必含板块只能多不能少。
- §D 功能需求详情必须按《物流费用账单 V2.10.3》模板逐功能点细写。

### 4.2 PRD 输出格式
- 固定输出为本地 Word 文档（.docx）。
- 图片必须正确渲染，且为之前生成的图片格式（SVG / PNG），不能是 Markdown 格式文件。
- PRD 中不得包含待确认标识和冗余信息。
- 所有待确认信息另外输出一个本地文档：`{需求名}-待确认项.docx`。

### 4.3 PRD 内容模块
必须包含（但不限于）：
1. 文档基本信息与修订记录
2. 背景、目标与 KPI
3. 已确认关键决策
4. 功能需求清单（FR）
5. 核心业务规则与限制
6. 业务流程与状态机
7. 需求影响范围
8. 涉及终端明细
9. 平台管理端功能需求详清
10. 客户端功能需求详清
11. 验收标准（AC）
12. 埋点需求
13. 非功能性需求
14. 边界设计检查清单
15. 关联需求
16. 待确认 / 开放问题

### 4.4 反问 Loop
- PRD 输出后进入循环确认。
- 颗粒度由粗到细：先反问流程维度，再反问细节如字段处理。
- 每个循环有问题都要积极反问，细化到字段级别。

---

## 五、GitHub 存档与评审后修改（Step 7-8）

### 5.1 首次存档
- 终版原型和终版 PRD 推送到**当前项目的 GitHub 仓库**等待评审。
- 推送前确保：
  - 仅包含结果文件
  - 过程文件已删除
  - 生成器脚本不上 GitHub
- **推送后门禁（必做）**：
  - 图片校验：打开 GitHub 文件页 / GitHub Pages 确认所有图片远端渲染正常（相对路径正确、图片已提交、无裂图），有问题按「图片校验 Loop」修复本地后重新推送。
  - **404 检查**：逐文件 `curl -I <raw或文件页URL>` 查 HTTP 状态，全部 `200`，无 `404`/`403`。
  - **预览页检查**：打开 GitHub Pages / raw 预览页，确认页面正常渲染、页内图片/CSS/JS 资源无 404；异常则修复本地后重推再查，循环到全过。

### 5.2 评审后修改
- 线下评审后，根据反馈下载 GitHub 文件到本地修改。
- 用户确认后再重新推送 GitHub 更新。
- 每次更新需记录修订记录。

---

## 六、收尾（Step 9）

### 6.1 文件整理
- 终版文件保存。
- 其他过程文件删除。
- 本地文件归入对应需求文件夹。
- 生成器脚本随源文件同目录保存，但不上 GitHub。

### 6.2 文档索引更新
- 更新项目文档索引（若项目有统一索引文件如 `00-文档索引.html` 则更新；否则在交付说明中给出文件清单）。

### 6.3 记忆更新
- 更新项目 MEMORY.md 或每日日志。

### 7.2 需求梳理图集

## 八、关键约束

1. **禁止臆测**：不确定数据必须 @干系人 占位或用 AskUserQuestion 反问。
2. **按需求分块**：每个需求独立文件夹，避免跨需求互相依赖。
3. **每次只展示最新产出**：present_files 只展示最新定版文件，不堆历史版本。
4. **PRD 先过结构检查**：所有 PRD/需求文档必须先调用 prd-structure-checker。
5. **图片格式正确**：PRD 中的图必须是图片格式，不能是 Markdown 流程图。
6. **图片校验 Loop**：含图交付物（图集/原型/平铺稿/PRD）生成后本地校验一次、推送 GitHub 后远端校验一次，循环到本地与远端图片全部正常。
7. **客户端设计原型必做**：客户端可交互原型确认后，必须额外产出无交互平铺、带一键复制源码的设计原型供墨刀使用（管理后台不需要）。
8. **产出校验 Hook（每步产出门禁）**：每个有交付物的 Step（2/3/4/5/6/7/8）产出后、进下一步前，必须查数量（不缺不重）+ 格式（类型/命名）+ 内容错乱（乱码/裂图/结构损坏/Markdown残留/依赖缺失）；Step 7/8 推送 GitHub 后额外查远端 404 + 预览页可打开。不通过立即修复重检，循环到全过，AI 须报告校验结果。详见第十节。

## 九、图片校验 Loop 详解

### 触发时机
- Step 2 图集、Step 3 原型/平铺稿、Step 5 PRD 生成或修改后（本地）。
- Step 7 / Step 8 推送 GitHub 之后（远端）。

### 校验清单
1. 每张图真实存在且成功嵌入/加载（无裂图、无红叉、无 404）。
2. 为图片格式（SVG/PNG/JPG），非 Markdown 流程图、代码块或文本占位。
3. 路径正确：本地相对路径可打开，GitHub 上同样可解析（注意大小写、层级）。
4. 内容匹配上下文，未放错图或用旧版本图。
5. Word/PDF 中图片清晰、未压扁或截断。

### 循环规则
- 本地：生成后立即自查（浏览器打开 HTML、转图检查 Word），修复到全部正常才推送。
- 远端：推送后打开 GitHub 文件页 / Pages 确认；裂图则修复本地后重推。
- 退出条件：本地与远端图片全部正常渲染。

## 十、产出校验 Hook 与 GitHub 远端检查

### 触发
- 每个有交付物的 Step（2/3/4/5/6/7/8）产出后、进下一步前，必须过门禁（详见 SKILL.md「产出校验 Hook」小节）。
- Step 7/8 推送 GitHub 后额外执行远端检查。

### 通用三查（本地）
1. **数量**：对照该步「本步交付物」清单，文件齐全、无遗漏、无重复副本、无过程文件混入。
2. **格式**：HTML 可解析无损坏；DOCX 合法 OOXML；命名符合规范；图片为真实格式非 Markdown。
3. **内容错乱**：① 乱码(U+FFFD) ② 裂图/缺图(src 空/占位/404) ③ 结构损坏(标签未配对/JS 报错/表格错位) ④ Markdown 残留 ⑤ 依赖缺失(外链 CSS/JS 失效)。

### 自动检查脚本
skill 附带 `produce-check-hook.py`，每步产出后可直接跑：
```bash
python references/produce-check-hook.py <需求文件夹> --step 2 --expect <需求名>-需求梳理图集.html
python references/produce-check-hook.py <需求文件夹> --step 5 --expect <需求名>-PRD.docx <需求名>-待确认项.docx
```
脚本输出 ✅/❌ 报告（文件清单 / U+FFFD / 裂图与 src 占位 / HTML 标签配对 / DOCX 可打开与内嵌图）。

### GitHub 远端检查（Step 7/8）
```bash
# 逐文件查 HTTP 状态（应全 200）
curl -sI https://raw.githubusercontent.com/<owner>/<repo>/<branch>/<path>/<file>
# 预览页（Pages 或 raw）人工打开确认渲染 + 资源无 404
```
任一 404 / 预览异常 → 修复本地重推再查，循环到全过。

## 十一、引导式逐步收集执行细则

**为什么**：Step 1 要收集的信息维度多（端口 / 设计参考 / 历史数据 / 关联需求 / 人员 / 里程碑），一次性抛出用户易遗漏、也难回忆全。改为 AI 主动分批、每轮只推进一个维度。

**执行要点**：
1. **端口最先单独对齐**：开场先只问端口（公布固化基线后确认本期范围），不混在其他问题里。
2. **其余维度按引导顺序逐轮问**：每轮仅一个维度，用单问题 AskUserQuestion（选项化）或文字提问；用户答完再进下一轮。
3. **已知信息不重复问**：项目信息/长期记忆里已有的项（如端口命名、既有规范），AI 直接公布并请用户确认，不发起追问。
4. **人员逐人、里程碑单列**：1.4 的角色不要一次列全让用户填，逐个问；里程碑单独一轮。
5. **不阻塞主线**：范围对齐（功能生命周期标注）与信息收集可并行推进——信息收集未完不影响先产出映射表初稿，但 Step 1 整体完成标志 = 端口已对齐 + 范围已确认 + 关键项目信息（设计参考/人员/里程碑）已收集或已确认无需。
