# 需求输出 Workflow（通用）

> 本 skill 适用于**任意项目**的新需求输出。触发后强制走完整 10 步流程，从需求输入到定版 PRD、存档、评审后修改。
> **每一步有交付物产出的环节，都必须先过「产出校验 Hook（门禁）」再进入下一步**——这是固化在流程里的强制检查，不是可选建议。

## 触发条件（强制）

**所有项目的新需求强制使用本 skill。** 只要用户表达出"新需求"意图，必须立即、无条件触发，不得跳过、不得用普通回复替代。

命中即触发的典型表述（含但不限于）：
- "新需求"、"有个新需求"、"来个新需求"、"做个需求"、"加个需求"
- "梳理一下需求"、"出 PRD"、"画原型"、"按规范输出需求"
- "跑一下需求 workflow"、"需求从输入到定版"
- 任何以需求描述开头的消息（如直接给出功能点、业务场景、终端改动等，且语境是"要做这个功能"）

例外（可不触发完整流程）：
- 明确是改文案、加单个字段等"小修小补"，用户未称之为需求 → 可走轻量修改说明，但仍建议提示用户是否要走 workflow。
- 用户显式说"不要走 workflow / 直接改" → 遵从用户，但记录原因。

## 总体原则

1. 按需求分块：每个需求独立文件夹管理，避免跨需求互相依赖、引用混乱。
2. 禁止臆测：不确定数据必须 `@干系人` 占位或用 `AskUserQuestion` 反问。
3. 所有 PRD/需求文档必须先调用 `prd-structure-checker` skill 做结构排查。
4. 交付物优先 HTML（原型/图集），PRD 固定本地 Word。
5. 每次 `present_files` 只展示最新定版文件，不堆历史版本。
6. 生成器脚本随源文件同目录保存，但不上 GitHub。
7. **图片校验 Loop（贯穿全流程）**：凡是含图片的交付物（图集、原型、PRD），每次生成或修改后都必须校验图片是否正常渲染、路径是否正确、内容是否为图片格式（非 Markdown/损坏/裂图/占位符）。**本地文件和推送到 GitHub 后都要各校验一次**——本地确认无误再推，推送后再拉取/打开确认远端渲染同样正常。发现问题立即修复并重新校验，循环直到本地与远端图片全部正常。详见「图片校验 Loop」小节。（注：图片校验是「产出校验 Hook」中"内容错乱-裂图"项的强化执行，不替代门禁。）
8. **产出校验 Hook（每步产出门禁）**：每个有交付物的 Step（2/3/4/5/6/7/8）在产出完成、进入下一步之前，必须执行固定检查——**数量**（不缺不重）+ **格式**（类型/命名正确）+ **内容错乱**（乱码/裂图/结构损坏/Markdown 残留/依赖缺失）；Step 7/8 推送 GitHub 后额外查**远端 404 + 预览页可打开**。不通过立即修复重检，循环到全过，不得跳过。AI 必须在回复中报告校验结果（✅/❌ + 具体项）。详见「产出校验 Hook（每步产出门禁）」小节。
9. **范围未确认，不出图集/FR（范围确认门禁，Step 1→Step 2 之间强制）**：进入 Step 2 产出「需求梳理图集」与功能需求清单(FR) 之前，必须完成范围确认——AI 在 Step 1 基于客户全生命周期 × 端口产出「功能生命周期映射表(初稿)」供用户逐行标注（本期做/增强/本期不做/待定 + 备注），**用户返回标注并确认本期范围后，才算范围锁定**；未确认前 AI 不得自动生成 Step 2 图集/FR，必须等待用户标注或显式确认。图集/FR 严格基于已确认范围，不得臆造未确认功能。详见「范围确认门禁」小节。
10. **引导式逐步收集（分轮逐项，禁止一次性抛出）**：Step 1 的多子项信息收集（设计参考 / 历史数据 / 关联需求 / 人员 / 里程碑等），AI **一次只推进一个维度**，用单问题 `AskUserQuestion` 或文字逐轮追问；用户答完当前项再问下一项。**严禁一次性把多个维度的问题打包抛出**——人脑容易遗漏、也无从下手。端口为最高优先、必须最先单独对齐（见 Step 1 端口基线），其余按 Step 1.1「引导顺序」依次推进。

## Workflow 概览

```
Step 0  任务判断与准备
Step 1  收集项目信息 → 写入项目参考/知识库 → 产出「功能生命周期映射表(初稿)」供用户标注
【范围确认门禁】用户逐行标注并确认本期范围后，才允许进入 Step 2
Step 2  需求梳理 → 输出统一 HTML 图集（严格基于「已确认范围」，不得超出）
Step 3  输出可交互原型（每端口）+ 客户端设计原型（平铺稿，墨刀用）
Step 4  PRD 结构检查
Step 5  PRD 输出（Word）+ 待确认项文档 + 反问 Loop
Step 6  本地路径确认
Step 7  终版存档 GitHub
Step 8  评审后本地修改 → 再推送
Step 9  收尾：归档、删过程文件、更新索引与记忆

贯穿：范围确认门禁（Step1 产出映射表 → 用户逐行标注确认范围 → 方进 Step 2；未确认 AI 不得生成图集/FR）
贯穿：产出校验 Hook（每步产出后立即执行：数量/格式/内容错乱；Step7/8 推送后查 404 + 预览页）
贯穿：图片校验 Loop（图集/原型/PRD 生成后 + 推送 GitHub 后各校验一次图片）
```

## 产出校验 Hook（每步产出门禁）

> **强制**。每一个有交付物产出的 Step（2/3/4/5/6/7/8），在产出写盘完成、进入下一步之前，必须执行本门禁。不通过 → 立即修复并重检 → 循环到全过，不得跳过。

### 触发与节奏
- **本地校验**：该 Step 产出文件写盘后立即执行（AI 用脚本或工具自查）。
- **远端校验**：仅 Step 7/8 推送 GitHub 后执行（查 404 + 预览页）。
- 每个 Step 末尾的「✅ 产出校验 Hook」小节列出该步专属检查项；通用三查对所有产出步生效。

### 通用三查（本地，所有产出步必做）
1. **数量（不缺不重）**：对照该 Step 规定的「本步交付物」清单，确认文件已产出且数量一致；无遗漏、无重复副本、无过程文件混入。
2. **格式（类型/命名正确）**：
   - HTML 可被解析、无 0 字节/损坏；DOCX 是合法 OOXML（python-docx 能打开）。
   - 文件名严格符合「文件命名规范」。
   - 图片为真实图片格式（SVG/PNG/JPG），非 Markdown 流程图/代码块/文本占位。
3. **内容错乱（五查）**：
   - ① 乱码：扫描 `U+FFFD` 替换符、异常编码、方块字。
   - ② 裂图/缺图：`<img>` 的 `src` 非空、非占位符、非 404；本地相对路径可打开（外链须确认可达）。
   - ③ 结构损坏：HTML 标签配对异常、JS 运行时报错、表格错位、样式崩。
   - ④ Markdown 残留：PRD/图集里的图应是图片，不得是 ```` ```mermaid ```` 或 Markdown 流程语法残留。
   - ⑤ 依赖缺失：外链 CSS/JS 失效导致页面样式/交互崩溃。

### GitHub 远端专属（Step 7/8，推送后必做）
1. **可达性（非 404）**：推送后逐文件调用 `curl -I` 或 WebFetch 查 HTTP 状态，全部 `200`，无 `404`/`403`。
2. **预览页可打开**：打开 GitHub Pages / raw 预览页，确认页面正常渲染、页内图片/CSS/JS 资源无 404。
3. 任一资源 404 或预览异常 → 修复本地后重新推送，再查，循环到全过。

### 自动检查方法
- 本地三查可跑 skill 附带脚本：`references/produce-check-hook.py <需求文件夹> [--step N] [--expect 文件名1 文件名2 ...]`
  （检查文件清单 / U+FFFD 乱码 / 裂图与 src 占位 / HTML 标签配对 / DOCX 可打开与内嵌图）。
- 远端检查用 `curl -sI <url>` 看 HTTP 状态，或 WebFetch 打开预览页人工确认渲染。
- AI 须在每步回复中报告校验结果（✅/❌ + 具体项），未报告视为未过门禁。

## Step 0: 任务判断与准备

1. 判断需求规模：
   - 小修小补（如改文案、加字段）：可跳过部分步骤，直接出修改说明。
   - 正常新需求：执行完整 workflow。
   - 大型中台/平台需求：拆分子模块分阶段输出。
2. 在 `{workspace}/.workbuddy/memory/YYYY-MM-DD.md` 创建或准备记录本需求工作。
3. 创建任务列表（TaskCreate），至少包含：项目信息收集、需求梳理、原型输出、PRD 输出、GitHub 存档。

## Step 1: 收集项目信息

### 1.1 必问信息（引导式逐项收集，禁止一次性抛出）

> ⚠️ **引导式收集规则**：本步所有信息**逐项、分轮收集**，AI 一次只推进**一个维度**，用单问题 `AskUserQuestion`（或文字）问当前项；用户答完再问下一项。**严禁一次性把下列所有问题打包丢给用户**——用户容易遗漏、也无从下手（本规则对应总体原则第 10 条）。

> **端口**为第一优先、必须最先单独对齐（见 Step 1 开头「端口基线」）；其余信息按下方「引导顺序」依次推进，每项单独一轮。

**引导顺序（每项单独一轮，逐项补齐）**：
1. **设计参考样式**（截图 / HTML / 墨刀 / Figma）—— 找不到合适参考必须追问，禁止凭空设计。
2. **历史数据 / 历史信息补充**（项目内已有则公布确认，不重复问）。
3. **关联需求与关联方式**（如有）。
4. **人员**（见 1.2，再拆 PM / 研发 / 测试 / UED / 数据 逐人问）。
5. **里程碑**。

> 若项目信息中已有某项，AI 直接公布并请用户确认即可，不重复提问；只有「未知且影响后续产出」的项才发起追问。

### 1.2 人员与里程碑（继续逐项引导）

人员按角色**逐人引导**（一次问一个角色，用户给完再问下一个）；里程碑最后单独一轮问。

- 产品负责人（PM）
- 研发负责人（前端 / 后端 / H5 / 客户端）
- 测试负责人
- UED / 交互
- 数据 / 埋点对接人
- 预计里程碑（单独一轮）

> 引导示例：先问「本轮先确认产品负责人(PM)是谁？」，用户答后下一轮问研发负责人，以此类推；避免一次列出 5 个角色让用户全填。

### 1.3 写入项目参考信息

将收集到的项目信息整理后：
- 写入本项目知识库或 `MEMORY.md`。
- 在后续步骤中直接引用，不再重复询问。

### 1.4 产出「功能生命周期映射表(初稿)」（范围对齐草稿，必做）

- **目的**：在产出图集/FR 前，先把"做什么"框清楚。AI 基于 **客户全生命周期**（如：认知触达→领取获取→卡包到账→适用浏览→下单抵扣→支付核销→售后退款→过期回收）与 Step 1 已确认端口，主动产出 **功能生命周期映射表(初稿)**。
- **结构**：每行 = 生命周期阶段 × 端口 的功能点；带「本期范围」标注位（本期做 / 增强 / 本期不做 / 待定）+ 备注框 + 导出标注结果按钮。功能点用徽章标来源（存量 / 新增? / 待确认）。
- **性质**：这是**对齐草稿，不是定稿**。产出后交用户逐行打标，AI 不替用户判定范围。
- **文件**：`{需求名}-功能生命周期标注.html`（命名规范见后）。
- 若用户直接口述核心功能点，AI 也可据此先结构化再补映射表；但必须有"用户标注/确认范围"这一步，不得跳过。

> ⚠️ Step 1 在用户**标注并确认范围前**不算完成；确认前不得进入 Step 2。

## 范围确认门禁（进入 Step 2 前的强制门禁）

> **强制**。在 Step 2 产出任何「需求梳理图集 / FR」之前，必须完成范围确认。本门禁卡的是 **INPUT（范围）是否锁定**，与「产出校验 Hook」卡 OUTPUT（质量）互补——两者缺一不可。

### 通过条件
- 用户已返回「功能生命周期映射表」标注结果（本期做/增强/不做/待定逐行确认），或；
- 用户已用其他方式**显式确认本期功能范围**（如直接列出本期做哪些功能点）。

### 不通过的处理
- 标注缺失 / 范围未确认 → AI **不得生成 Step 2 图集/FR**，必须先用 `AskUserQuestion` 或等待用户标注返回；可主动催办并提示"标完点导出"。
- 用户标注中出现「待定/待确认」行 → 这些行在 Step 2 中归入「待确认项」，不得作为已确认功能写进 FR。

### 确认后动作
- 将「已确认范围」整理为需求框架/知识库输入，作为 Step 2 图集与 FR 的**唯一依据**；图集/FR 严格基于已确认范围，未确认功能一律不写。
- AI 在确认后应在标注文件写入 `SCOPE_CONFIRMED` 标记，或在同目录生成 `{需求名}-范围已确认.md` 伴随文件，供 `produce-check-hook.py --scope` 机器校验。

### 机器辅助校验
- `produce-check-hook.py --scope <功能生命周期标注.html>`：进入 Step 2 前检查该标注文件已存在且含确认标记（SCOPE_CONFIRMED 或伴随确认文件）；缺失则门禁不通过。

## Step 2: 需求梳理

> ⚠️ **仅当「范围确认门禁」通过后才能进入本步**。图集与 FR 严格基于 Step 1 已确认的本期范围，不得超出或臆造未确认功能。

输出以下内容到一个统一 HTML 文件：`{需求名}-需求梳理图集.html`。

### 2.1 背景与目标
- 业务背景、业务目标、用户目标。
- 可量化 KPI / 成功指标。

### 2.2 已确认关键决策
- 用表格列出决策点与结论。

### 2.3 需求影响范围
- 用表格说明对每个端口的页面和历史功能影响。
- 说明对关联需求、数据、中台能力、消息中心等的影响。

### 2.4 业务流程图
- 必须包含主流程 + 异常/分支流程。
- 必须为常用流程图样式（SVG / 流程图图片），禁止 Markdown 流程图。

### 2.5 信息架构
- 清晰界定前端展示层 / 后端配置层 / 数据层的映射关系。
- 必须为图片 / SVG，禁止 Markdown 格式。

### 2.6 状态机与状态流转规则
- 凡涉及状态变化的实体（订单、支付、卡券、任务等），提供状态流转表。
- 格式：起始状态 + 触发动作(Action) + 前置/后置条件 = 目标状态。

### 2.7 用户旅程 / 脑图
- 根据业务流程和用户生命周期，关联板块确认功能范围。
- 脑图优先输出 Markdown 大纲（可导入钉钉文档切换为思维导图）。

### 2.8 功能需求清单（FR）
- 按模块拆解功能需求，编号便于追溯。
- 为后续 PRD 的 §四 功能需求详情做准备。

### 2.9 交付与确认
- 生成统一 HTML 文件：`{需求名}-需求梳理图集.html`。
- 用 `present_files` 展示给用户确认。
- 用户确认后保留到需求文档中；未确认则迭代更新同一文件。

### 2.10 ✅ 产出校验 Hook
- 数量：仅 `{需求名}-需求梳理图集.html` 一个文件（不混入原型/PRD/过程稿）；可跑 `produce-check-hook.py <dir> --step 2 --expect <需求名>-需求梳理图集.html`。
- 格式：HTML 可解析、命名合规；2.1–2.8 各板块齐全。
- 内容错乱：无 U+FFFD；流程图/信息架构/状态机为 SVG 图片而非 Markdown 残留；无裂图；标签配对正常。

## Step 3: 输出可交互原型

### 3.1 按端口输出

根据 Step 1 了解的端口，为每个端口输出 HTML 格式可交互原型：
- 管理后台：列表页、表单页、弹窗、状态切换、空态。
- 客户端：页面流程、按钮/入口、弹窗、异常态、空态。
- H5：页面流程、与原生 App 的交互点。

### 3.2 原型确认清单

每次交付原型时，用以下清单与用户确认：
- 页面是否完整？
- 弹窗是否完整？
- 交互逻辑是否完整？
- 异常分支是否覆盖？
- 字段 / 按钮 / 权限是否完整？

### 3.3 客户端设计原型（平铺稿，墨刀用）【必做】

可交互原型确认后，**仅针对客户端**（App / H5 / 小程序，不含管理后台）额外输出一份「设计原型」平铺稿，供用户复制到墨刀：
- **平铺展示**：所有客户端页面按顺序平铺在一个 HTML 页里，**无交互、无跳转、无弹窗逻辑**，纯静态视图，一屏可总览全部页面。
- **一键复制源码**：页面上提供「一键复制源码」按钮，点击即把该原型的完整 HTML 源码复制到剪贴板，方便用户粘贴到墨刀。
- 每个页面配简短标题标注（页面名 / 用途），但不做任何交互控件。
- 文件命名：`{需求名}-客户端设计原型-flat.html`。
- 用 `present_files` 展示给用户确认。

### 3.4 文件命名

- 单页原型：`{需求名}-{模块}-{页面}.html`
- 合并交互原型：`{需求名}-{模块}-合并交互原型.html`
- 客户端设计原型（平铺稿）：`{需求名}-客户端设计原型-flat.html`

### 3.5 图片校验

原型/平铺稿若引用图片（截图、图标、切图），生成后按「图片校验 Loop」逐一确认本地渲染正常；进入 Step 7 推送后需再次确认远端渲染正常。

### 3.6 ✅ 产出校验 Hook
- 数量：各端口原型文件数量与本步端口一致；客户端平铺稿恰 1 个；无重复副本。
- 格式：HTML 可解析、命名合规；平铺稿含「一键复制源码」按钮。
- 内容错乱：浏览器打开可交互原型无 JS 运行时报错；引用图片无裂；无 Markdown 残留；外链 CSS/JS（如有）未失效。

## Step 4: PRD 结构检查

在输出 PRD 前，必须：
1. 调用 `prd-structure-checker` skill 对 PRD 结构做排查。
2. 确保必含板块只能多不能少。
3. 对缺失或未知内容直接反问用户补充，禁止臆造。
4. 通过结构检查后再进入 Step 5。

> **结构权威**：PRD 终稿板块骨架以 `prd-structure-checker` 的「一~九」为准（对齐《PRD 输出规范与标准示例（修订于260715）》），**严禁按旧「A~I」写法产出**；§四 功能需求详情须按「页面」拆解（X.1 原型与交互 + X.2 详细逻辑）。

### 4.1 ✅ 产出校验 Hook（结构门禁）
- 本步门禁 = `prd-structure-checker` 必须被调用且输出"通过"（板块只能多不能少、§四 深度达标）。
- 内容错乱：缺失 / 未知项已反问或 `⚠️ 待确认 @干系人` 占位，无臆造；无 Markdown 流程图残留。

## Step 5: PRD 输出与反问 Loop

### 5.1 输出格式

- PRD 固定输出为本地 Word 文档：`{需求名}-PRD.docx`。
- 图片必须正确渲染，且为 Step 2 生成的图片格式（SVG / PNG），不能是 Markdown 格式文件。
- PRD 中不得包含待确认标识和冗余信息。
- 所有待确认信息另外输出一个文档：`{需求名}-待确认项.docx`。

### 5.2 PRD 必含内容

> **结构权威**：PRD 终稿以 `prd-structure-checker` 的「一~九」为唯一骨架。以下 16 项为**内容提示清单**，须归入对应板块（功能需求与两端详清→§四、埋点需求→§五、验收标准→§六、边界清单→§七、待确认→§九），不得另起一套编号。

1. 文档基本信息与修订记录
2. 背景、目标与 KPI
3. 已确认关键决策
4. 功能需求清单（FR）
5. 核心业务规则与限制
6. 业务流程与状态机
7. 需求影响范围
8. 涉及终端明细
9. 平台管理端功能需求详清
10. 客户端功能需求详清
11. 验收标准（AC）
12. 埋点需求：必须引用《泰小虎埋点表v2.3》规范（见 `prd-structure-checker/references/tracking-spec.md`；权威源文件 `D:/梯子/泰小虎埋点表v2.3 (5).xlsx`，在线方案 https://ddyuan-spec.github.io/taixiaohu/taixiaohu-tracking-plan.html ）。事件与属性**只能取既有目录（30 个事件 / L1-L3），禁止凭空编造**；新事件须走埋点评审，不在 PRD 内新增定义；引用埋点参数一律用友盟后台中文名。
13. 非功能性需求
14. 边界设计检查清单
15. 关联需求
16. 待确认 / 开放问题

### 5.3 §四 功能需求详情标准

按《物流费用账单 V2.10.3》模板逐功能点细写：
- 一个按钮 / 表单 = 一个独立小节。
- 含功能说明、显示位置、权限、前置条件、逐步操作逻辑（异常分支 + 校验提示文案）。
- 列表字段表（字段名称 / 需求说明 / 备注 + 排序分页合计唯一性）。
- 编辑表单输入限制（必填 / 类型 / 长度 / 小数位）+ 计算公式 + 数据演练。
- 查询条件规格、字段级数据来源。
- 逐页截图或原型引用。

### 5.4 反问 Loop

PRD 输出后进入循环确认：
1. 先反问流程维度：整体流程是否完整？模块拆分是否合理？
2. 再反问细节：字段处理、数据来源、计算公式、权限、异常分支。
3. 用户提出问题后，修改 PRD 并重新确认。
4. 循环直到用户确认 PRD 无问题。

### 5.5 ✅ 产出校验 Hook
- 数量：PRD.docx + 待确认项.docx 两个文件，不缺、不重。
- 格式：docx 可被 python-docx 打开（OOXML 合法）；命名合规（可跑 `produce-check-hook.py <dir> --step 5 --expect <需求名>-PRD.docx <需求名>-待确认项.docx`）。
- 内容错乱：无 U+FFFD 乱码；图片已**内嵌**（非外链，避免远端裂图）；PRD 中无待确认标识残留；§四 逐功能点齐全；无 Markdown 流程图残留。

## Step 7: 终版存档 GitHub

1. 确保存档内容仅包含终版原型和终版 PRD。
2. 删除过程文件：临时副本、verify_*.js、中间版本 HTML 等。
3. 生成器脚本保留在本地同目录，但不上 GitHub。
4. 推送到**当前项目的 GitHub 仓库**等待评审。
5. 更新项目文档索引（若项目有统一索引文件，如 `00-文档索引.html`，则更新；否则在交付说明中给出完整文件清单）。

### 7.1 ✅ 产出校验 Hook（含远端门禁）
- 本地：先按通用三查确认终版文件无误；且过程文件已删、生成器脚本未上 GitHub。
- 远端（推送后**强制**）：
  1. 逐文件 `curl -I <raw或文件页URL>` 查 HTTP 状态，全部 `200`，无 `404`/`403`。
  2. 打开 GitHub Pages / raw 预览页，确认页面正常渲染、页内图片/CSS/JS 资源无 404。
  3. 任一资源 404 或预览异常 → 修复本地后重新推送，再查，循环到全过。
- AI 必须在回复中报告：本地三查 ✅/❌ + 远端 404 检查 ✅/❌ + 预览页结论。

## Step 8: 评审后本地修改

3. 用 `present_files` 展示修改后的文件给用户确认。
4. 用户确认无问题后，重新推送 GitHub 更新。
5. 每次更新需记录修订记录。

### 8.1 ✅ 产出校验 Hook
- 本地：修改后重跑通用三查（数量/格式/内容错乱）。
- 远端：重新推送后再次查 404 + 预览页，循环到全过。
