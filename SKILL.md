# 泰小虎需求输出 Workflow

## 触发条件（强制）

**本项目（泰小虎）所有需求强制使用本 skill。** 只要用户表达出"新需求"意图，必须立即、无条件触发，不得跳过、不得用普通回复替代。

命中即触发的典型表述（含但不限于）：
- "新需求"、"有个新需求"、"来个新需求"、"做个需求"、"加个需求"
- "梳理一下需求"、"出 PRD"、"画原型"、"按规范输出需求"
- "跑一下需求 workflow"、"需求从输入到定版"
- 任何以需求描述开头的消息（如直接给出功能点、业务场景、终端改动等，且语境是"要做这个功能"）

例外（可不触发完整流程）：
- 明确是改文案、加单个字段等"小修小补"，用户未称之为需求 → 可走轻量修改说明，但仍建议提示用户是否要走 workflow。
- 用户显式说"不要走 workflow / 直接改" → 遵从用户，但记录原因。

## 总体原则

1. 按需求分块：每个需求独立文件夹，禁止跨需求文件夹引用/依赖。
2. 禁止臆测：不确定数据必须 `@干系人` 占位或用 `AskUserQuestion` 反问。
3. 所有 PRD/需求文档必须先调用 `prd-structure-checker` skill 做结构排查。
4. 交付物优先 HTML（原型/图集），PRD 固定本地 Word。
5. 每次 `present_files` 只展示最新定版文件，不堆历史版本。
6. 生成器脚本随源文件同目录保存，但不上 GitHub。

## Workflow 概览

```
Step 0  任务判断与准备
Step 1  收集项目信息 → 写入项目参考/知识库
Step 2  需求梳理 → 输出统一 HTML 图集
Step 3  输出可交互原型（每端口）
Step 4  PRD 结构检查
Step 5  PRD 输出（Word）+ 待确认项文档 + 反问 Loop
Step 6  本地路径确认
Step 7  终版存档 GitHub
Step 8  评审后本地修改 → 再推送
Step 9  收尾：归档、删过程文件、更新索引与记忆
```

---

## Step 0: 任务判断与准备

1. 判断需求规模：
   - 小修小补（如改文案、加字段）：可跳过部分步骤，直接出修改说明。
   - 正常新需求：执行完整 workflow。
   - 大型中台需求：拆分子模块分阶段输出。
2. 在 `{workspace}/.workbuddy/memory/YYYY-MM-DD.md` 创建或准备记录本需求工作。
3. 创建任务列表（TaskCreate），至少包含：项目信息收集、需求梳理、原型输出、PRD 输出、GitHub 存档。

---

## Step 1: 收集项目信息

### 1.1 必问信息

向用户询问以下信息；若项目信息中已有，则公布并确认即可：
- 是否有管理后台？有几个？分别怎么命名？
- 是否有客户端？有几个？分别是什么？怎么命名？（App / 小程序 / H5 / PC）
- 本期需求涉及哪些端口？
- 是否有设计参考样式？（截图 / HTML / 墨刀 / Figma）
  - 找不到合适参考时必须追问补充，禁止凭空设计。
- 是否有历史数据或历史信息需要补充？
- 是否关联其他需求？关联方式是什么？

### 1.2 人员与里程碑

- 产品负责人（PM）
- 研发负责人（前端 / 后端 / H5 / 客户端）
- 测试负责人
- UED / 交互
- 数据 / 埋点对接人
- 预计里程碑

### 1.3 写入项目参考信息

将收集到的项目信息整理后：
- 写入本项目知识库或 `MEMORY.md`。
- 在后续步骤中直接引用，不再重复询问。

---

## Step 2: 需求梳理

输出以下内容到一个统一 HTML 文件：`{需求名}-需求梳理图集.html`。

### 2.1 背景与目标
- 业务背景、业务目标、用户目标。
- 可量化 KPI / 成功指标。

### 2.2 已确认关键决策
- 用表格列出决策点与结论。

### 2.3 需求影响范围
- 用表格说明对每个端口的页面和历史功能影响。
- 说明对关联需求、数据、ERP、消息中心等的影响。

### 2.4 业务流程图
- 必须包含主流程 + 异常/分支流程。
- 必须为常用流程图样式（SVG / 流程图图片），禁止 Markdown 流程图。

### 2.5 信息架构
- 清晰界定前端展示层 / 后端配置层 / 数据层的映射关系。
- 必须为图片 / SVG，禁止 Markdown 格式。

### 2.6 状态机与状态流转规则
- 凡涉及状态变化的实体（订单、券、支付等），提供状态流转表。
- 格式：起始状态 + 触发动作(Action) + 前置/后置条件 = 目标状态。

### 2.7 用户旅程 / 脑图
- 根据业务流程和用户生命周期，关联板块确认功能范围。
- 脑图优先输出 Markdown 大纲（可导入钉钉文档切换为思维导图）。

### 2.8 功能需求清单（FR）
- 按模块拆解功能需求，编号便于追溯。
- 为后续 PRD 的 §D 功能需求详情做准备。

### 2.9 交付与确认
- 生成统一 HTML 文件：`{需求名}-需求梳理图集.html`。
- 用 `present_files` 展示给用户确认。
- 用户确认后保留到需求文档中；未确认则迭代更新同一文件。

---

## Step 3: 输出可交互原型

### 3.1 按端口输出

根据 Step 1 了解的端口，为每个端口输出 HTML 格式可交互原型：
- 管理后台：列表页、表单页、弹窗、状态切换、空态。
- 客户端：页面流程、按钮/入口、弹窗、异常态、空态。
- H5：页面流程、与原生 App 的交互点。

### 3.2 原型确认清单

每次交付原型时，用以下清单与用户确认：
- 页面是否完整？
- 弹窗是否完整？
- 交互逻辑是否完整？
- 异常分支是否覆盖？
- 字段 / 按钮 / 权限是否完整？
- 是否需要平铺稿（墨刀用）？

### 3.3 文件命名

- 单页原型：`{需求名}-{模块}-{页面}.html`
- 合并交互原型：`{需求名}-{模块}-合并交互原型.html`
- 平铺稿：`{需求名}-{页面}-flat.html`

---

## Step 4: PRD 结构检查

在输出 PRD 前，必须：
1. 调用 `prd-structure-checker` skill 对 PRD 结构做排查。
2. 确保必含板块只能多不能少。
3. 对缺失或未知内容直接反问用户补充，禁止臆造。
4. 通过结构检查后再进入 Step 5。

---

## Step 5: PRD 输出与反问 Loop

### 5.1 输出格式

- PRD 固定输出为本地 Word 文档：`{需求名}-PRD.docx`。
- 图片必须正确渲染，且为 Step 2 生成的图片格式（SVG / PNG），不能是 Markdown 格式文件。
- PRD 中不得包含待确认标识和冗余信息。
- 所有待确认信息另外输出一个文档：`{需求名}-待确认项.docx`。

### 5.2 PRD 必含内容

1. 文档基本信息与修订记录
2. 背景、目标与 KPI
3. 已确认关键决策
4. 功能需求清单（FR）
5. 核心业务规则与限制
6. 业务流程与状态机
7. 需求影响范围
8. 涉及终端明细
9. 平台管理端功能需求详清
10. 客户端功能需求详清
11. 验收标准（AC）
12. 埋点需求
13. 非功能性需求
14. 边界设计检查清单
15. 关联需求
16. 待确认 / 开放问题

### 5.3 §D 功能需求详情标准

按《物流费用账单 V2.10.3》模板逐功能点细写：
- 一个按钮 / 表单 = 一个独立小节。
- 含功能说明、显示位置、权限、前置条件、逐步操作逻辑（异常分支 + 校验提示文案）。
- 列表字段表（字段名称 / 需求说明 / 备注 + 排序分页合计唯一性）。
- 编辑表单输入限制（必填 / 类型 / 长度 / 小数位）+ 计算公式 + 数据演练。
- 查询条件规格、字段级数据来源。
- 逐页截图或原型引用。

### 5.4 反问 Loop

PRD 输出后进入循环确认：
1. 先反问流程维度：整体流程是否完整？模块拆分是否合理？
2. 再反问细节：字段处理、数据来源、计算公式、权限、异常分支。
3. 用户提出问题后，修改 PRD 并重新确认。
4. 循环直到用户确认 PRD 无问题。

---

## Step 6: 本地路径确认

1. 输出 PRD 和待确认项文档的本地完整路径。
2. 等待用户确认。
3. 用户确认有问题则返回 Step 5；确认无问题则进入 Step 7。

---

## Step 7: 终版存档 GitHub

1. 确保存档内容仅包含终版原型和终版 PRD。
2. 删除过程文件：临时副本、verify_*.js、中间版本 HTML 等。
3. 生成器脚本保留在本地同目录，但不上 GitHub。
4. 推送到 GitHub 等待评审。
5. 更新 `00-文档索引.html`，加入最新产出链接。

---

## Step 8: 评审后本地修改

1. 用户线下评审后，根据反馈下载 GitHub 文件到本地。
2. 在本地修改原型和 PRD。
3. 用 `present_files` 展示修改后的文件给用户确认。
4. 用户确认无问题后，重新推送 GitHub 更新。
5. 每次更新需记录修订记录。

---

## Step 9: 收尾

### 9.1 文件整理
- 终版文件保存到需求文件夹。
- 删除其他过程文件。
- 本地文件归入文档。

### 9.2 文档索引更新
- 更新 `00-文档索引.html`。

### 9.3 记忆更新
- 在 `{workspace}/.workbuddy/memory/YYYY-MM-DD.md` 追加本需求完成记录。
- 对于长期项目规则或用户偏好，更新 `{workspace}/.workbuddy/memory/MEMORY.md` 或 `~/.workbuddy/MEMORY.md`。

### 9.4 任务关闭
- 用 TaskUpdate 将所有相关任务标记为 completed。

## 注意事项

- 本 workflow 适用于泰小虎项目的新需求输出；小修小补可酌情裁剪步骤。
- 每个需求必须独立文件夹，禁止跨需求引用。
- 任何不确定数据禁止臆测，必须反问或占位。
- PRD 必须先过 `prd-structure-checker` skill。
- 每次 `present_files` 只展示最新定版文件。
